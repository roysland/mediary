# Improvements Backlog

This file lists concrete improvement tasks for the project without changing code. It is intended as a planning document for future work.

## Scope and constraints

- Project type: Go server-rendered web app with SQLite, html templates, HTMX, and browser-side JavaScript.
- Important rule: `internal/db/` is generated by `sqlc` and must not be edited directly.
- For database changes, edit `db/schema.sql` and `db/queries.sql`, then run `sqlc generate`.
- Current auth is a development stub in `internal/auth/auth.go`.

## Priority order

1. Fix naming and routing inconsistencies.
2. Extract duplicated handler and request logic.
3. Move inline JS and CSS out of templates.
4. Clean up schema and query duplication.
5. Add tests around handlers and settings logic.
6. Separate startup, config, and migration responsibilities.

## 8. Introduce explicit configuration instead of hardcoded runtime values

```bash

```

## 26. Add a deletion strategy for entries or remove the unfinished UI
✅ **COMPLETED**

**Implementation details:**

- Route: `POST /entry/{id}/delete` (registered in `internal/server/routers.go`)
- Handler: `deleteEntry()` in `internal/server/entries.go` - validates user ownership via `requireUserID()`
- Database: `DeleteEntry` query in `db/queries.sql` 
- Cascade handling: SQLite foreign key constraints with `ON DELETE CASCADE` automatically remove related `trackable_values`
- JavaScript: `deleteEntry()` function in `web/static/entries.js` - confirms deletion and updates DOM on success
- UI: `.delete-note-button` in `internal/views/entries.html` - user-facing trigger

**Strategy chosen:**

- Hard delete of entries (not soft delete)
- Automatic cascade deletion of all associated trackable values
- This is the correct approach for this domain (deletion is permanent and complete)
