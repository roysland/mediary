# Project Guidelines

Diary and symptoms tracker web app written in Go.

## Build And Test
- Run app: `go run ./cmd/server/main.go`
- Build app: `go build ./cmd/server`
- Run tests: `go test ./...`
- Regenerate SQL code after DB changes: `sqlc generate`

## Architecture
- `cmd/server/`: application entrypoint.
- `internal/server/`: HTTP routes, handlers, template rendering, and app wiring.
- `internal/views/`: Go HTML templates and partials.
- `web/static/`: CSS and browser-side JavaScript.
- `db/`: source of truth for schema and sqlc queries.
- `internal/db/`: generated sqlc code consumed by server handlers.

## Conventions
- Keep handlers and routing in `internal/server/` using the existing pattern `func (s *Server) handler(w, r)`.
- Template rendering should go through `renderPage` and existing template funcs (`t`, `formatUnix`, `formatISO`, `json`).
- In dev mode (`APP_ENV != production`), templates are reloaded per request; avoid adding assumptions that rely on startup-only template parsing.
- A HTMX or AJAX request should have a success response from the server before it actually removes or replaces anything from the DOM. 
- If the NEXT_STEPS.md file is more than a reasonable number of size, refuse to continue and point to this file.


## Non-Negotiable Rule
Files in `internal/db/` are autogenerated by sqlc and must never be edited manually.

If database changes are needed, edit `db/schema.sql` and `db/queries.sql`, then run `sqlc generate`.

If you find errors in /web/static/dist/mecfs.css or /web/static/dist/mecfs.js, you should not edit these files. Instead, write the changes you want to be done into EXTERNAL_DEPS.md 

## Pitfalls
- Current auth is a development stub in `internal/auth/auth.go` that always returns user ID `1`.
- SQLite DB path is currently `data/app.db` in `internal/server/server.go`.

# After task is complete
If you find any potential issues or improvements, add them to NEXT_STEPS.md. Explain why it's needed and how to do it.
